# rapid-mlx 0.10.x TODO — top-tier for the 3 Mac families

Derived from 4-agent parallel research (Qwen 3.6 + Gemma 4 + gpt-oss externals + rapid-mlx internal audit) on 2026-07-04, restructured 2026-07-04 per operator directive into a **versioned patch-release cadence** rather than a single monolithic release. Scope pivoted from the earlier quantization-first plan (SpinQuant / W8A8 / QuIP#) → **top-tier support for the three model families that Mac users actually run**. Quant axis moved to `0.11-TODO.md`.

> **Read me first — the big finding.** Qwen 3.6 has a **native MTP head baked into every checkpoint** (27B + 35B-A3B), sharing the embedding + `lm_head` weights with the main model. No external drafter needed. Our `0.9.x` profile note "no MTP/drafter trained" is **wrong** and needs correcting in 0.10.3. mlx-community already mirrors `Qwen3.6-27B-MTP-4bit` and `Qwen3.6-35B-A3B-MTP-4bit` — the assets are ready. Reference in-the-wild speedup: **2.24× decode on 27B** (`youssofal/MTPLX`, MLX-native).

> **Read me second — Gemma 4 KV-sharing is missing.** Google published assistant sidecars for **all 5 sizes** (E2B, E4B, 12B, 26B-A4B, 31B) with KV-sharing borrowed from Gemma 3n — the drafter reuses the target's KV cache so it skips prefill. Our 0.9.13 shipped 12B only, and does NOT use KV-sharing. Every turn pays for drafter prefill. Fixing this is the biggest single perf win of 0.10 — lands in 0.10.3 (cascaded 2026-07-05, was 0.10.2 before 0.10.1 was consumed by the Gemma 4 fresh-install vendor fix).

> **Read me third — gpt-oss EAGLE-3 is deferred, not shipped.** Amazon / RedHat / NVIDIA published EAGLE-3 drafters for gpt-oss-20b and 120b, but ALL are CUDA / vLLM-only. MLX port = weeks of greenfield kernel work with no upstream precedent. 0.10 gpt-oss scope is aliases + `reasoning_effort` passthrough + harmony parser hardening. EAGLE-3 goes on the 0.11+ watchlist.

> **Read me fourth — the SOTA vs adhoc gate.** Every Layer B (perf) item on this file must be paper-backed (arxiv ID or major-lab publication) AND lossless-preserving. No community "special quants" without formal treatment. No layer-skip / early-exit heuristics without paper. No "found on Twitter" claims. The list in each version below has been (or will be — see research pending) filtered against this gate. See `## Perf technique SOTA gate` at the bottom.

> **Read me fifth — spec-decode interface freeze (2026-07-04).** PR #1012 landed a vLLM-style `--speculative-config` frontend + plugin registry at `vllm_mlx/spec_decode/{config.py,registry.py}`. **Every 0.10.x spec-decode landing MUST plug into this interface — do NOT add new top-level CLI flags** (`--enable-<method>`, `--<method>-drafter-path`, etc. — those are legacy fallbacks). Contract:
>
> - Register the method in `registry.py` via `register_spec_decoder(SpecDecoderPlugin(method="...", description="...", config_enabled=False, legacy_hint="..."))`.
> - Wire the backend behind a common runner interface (interface TBD — first migrator defines it).
> - Flip `config_enabled=True` when the backend is wired, and remove the legacy flag once no code path still hits it.
> - Until wired, `--speculative-config '{"method":"..."}' rejects early with `legacy_hint`. This is intentional.
>
> Currently registered (all `config_enabled=False`): `dflash`, `mtp`, `suffix`/`ngram`. **Explicit mapping to 0.10.x work** (cascaded 2026-07-05 per odd/even convention):
> - **0.10.4** Google MTP sidecars all 5 sizes + Gemma-3n KV-share loader → migrate `method=mtp` backend, flip `config_enabled=True`.
> - **0.10.6** Qwen 3.6 native MTP + FastMTP head → same `method=mtp` runner (unified with Gemma 4). `qwen3_6_inject.py` plugs into the runner, not into a new flag.
> - **0.10.8 (candidate)** gpt-oss Speculative MoE → `register_spec_decoder(SpecDecoderPlugin(method="spec_moe", ...))` — new registration, same interface.
> - **0.11 (watchlist)** EAGLE-3 → `register_spec_decoder(SpecDecoderPlugin(method="eagle3", ...))`.
>
> First 0.10.x PR that touches spec-decode MUST also define the common runner interface so subsequent PRs plug in cleanly. Do NOT bikeshed the interface via 5 PRs; land the first migration + interface together.

## 0.10.4 Release acceptance — Gemma 4 perf focus (feature)

Layer B for the family where our shipped 0.9.13 baseline has the largest gap. KV-sharing is the biggest single perf win of the entire 0.10 cycle.

### The five items (SOTA-gate-approved, finalized 2026-07-04)

- [ ] **[P0 · Tier A1] Gemma-3n cross-layer KV sharing loader plumb.** Middle-block K/V shared to all top layers per Gemma 3n architecture → **2× prefill / TTFT** (Google-benchmarked Gemma-4 vs Gemma-3-4B). Lossless (weight-declared). Blocker today: mlx-lm's KV dataclass allocates per-layer and silently duplicates the middle-block state. Fix is pure Python via `make_prompt_cache` borrow-mode; no Metal kernel work needed. This is the **highest ROI single item in the entire 0.10 cycle** — do this first. Effort: ~1 week. **Paper references**: Google Gemma-3 report (arxiv 2503.19786) + KV-sharing systematic study (arxiv 2410.14442) + Gemma 3n developer guide.
- [ ] **[P0 · Tier A / family pick] Google MTP assistant sidecars for E2B / E4B / 26B-A4B / 31B.** Extend `vllm_mlx/spec_decode/mtp/gemma4_inject.py` to each size's drafter shape (4 layers × per-size hidden). Load `-assistant-bf16` weights via existing sidecar path. Bench each vs baseline: expect +25-30% structured tok/s matching the 12B result. Effort: ~1 week. **Production reference**: Ollama v0.23.1 + llama.cpp both ship this — we're catching up. `research_ollama_gemma4_mtp.md` in knowledge memory has the K controller notes.
- [ ] **[P1 · Tier A2] Fused qSDPA + RoPE + RMSNorm Metal kernel (mlx-qsdpa integration).** Per mlx-qsdpa (Thump604): **1.7× decode vs mlx-lm baseline + 4× KV mem via inline dequant**. Lossless. Partial land in mlx-lm HEAD; ZMLX / mlx-metal-kernels are incubators for the rest. Wiring this in also unblocks the deferred TurboQuant K4V4 (Tier B2). Effort: ~1-2 weeks. **Paper reference**: mlx-qsdpa / mlx-metal-kernels repo READMEs; upstream mlx-lm PR tracking the split-fusion.
- [ ] **[P1 · Tier B2, gated on A2] TurboQuant / PolarQuant K4V4 KV cache.** arxiv 2504.19874 — 6× mem, 8× attention on H100, data-oblivious, ppl-neutral. Blocker: mlx-lm draft PR #1059 is **0.5× vs FP16 without fused Metal dequant-matmul** (same kernel scaffolding as A2). **Ship in the same PR pair as A2 or roll to 0.10.4 if A2 slips.** Do NOT ship without the fused kernel — will regress default users.
- [ ] **[P2 · defensive] PLE quantization probe test.** Early mlx-community/unsloth 4-bit variants quantized Per-Layer-Embedding tables → garbage output. Current mlx-community re-quants keep PLE in bf16 (probe verifies). Add to acceptance dashboard so bug can't recur silently. Effort: ~0.5 day. **Note**: this is a regression guard, not a perf technique — but it's cheap and matches the family focus.

### Perf-technique SOTA gate — Gemma 4 verdict

All five items above cleared the gate (A1 = paper + Google prod + lossless; A2 = incubator kernel + measured 1.7× on Gemma 4; MTP sidecar family = Ollama/llama.cpp prod + Gemma 4 12B already lossless in 0.9.13; B2 = paper + gated on A2; PLE probe = regression guard, not a claim). Zero Tier-C adhoc items in this list.

**Rejected during synthesis** (would have naively fit here but flunked the gate):
- KV eviction (H2O / SnapKV / StreamingLLM sinks) — approximate, fails quality gate (H2O collapses to 38% RULER, SnapKV misses 90% NIAH at claimed compression, StreamingLLM forgets middle content by design)
- Expert prefetch family (ExpertFlow / Fiddler / MoE-Infinity / PowerInfer-2 / Pregated MoE) — all solve CPU→GPU PCIe copy which doesn't exist on Apple Silicon UMA
- Continuous batching / PagedAttention / FlashInfer sparse prefill — multi-tenant tricks; rapid-mlx is single-user

## Perf technique SOTA gate — the standing rules

Every Layer B item must satisfy ALL of the following before shipping:

1. **Paper-backed** — arxiv ID or major-lab (Google / Meta / MS / OpenAI / Anthropic / Apple / Alibaba / DeepSeek) publication. "Some Discord post" doesn't count.
2. **Verified quality preservation** — either provably lossless-in-distribution (e.g., MTP, EAGLE-3 with acceptance verification) or measurable-and-acceptable quality delta (e.g., int8 W8A8 with < 2 point AIME25 regression as documented gate). No "seems fine" without a bench row.
3. **Production-deployed at scale** — vLLM / TensorRT-LLM / Ollama / mlx-lm / Google Vertex / OpenAI production shipping it counts. Discord-only doesn't.
4. **Fit for our chain-decode / MTP-adjacent workload** — tree-attention experiments that would require rewriting our batched engine are out-of-scope for 0.10.
5. **Applicable to at least one of the three target families** — no generic optimizations that don't move any hero alias.

### Adhoc / "民科" REJECT list (a running list — extend as we encounter them)

- Random temperature / top-p tuning sold as "perf" — that's quality, not speed
- Community quants without paper backing (e.g. "special calibration set")
- Layer-skipping / early-exit heuristics without formal treatment
- "Found on Twitter" claims of Nx speedup that no one has reproduced
- Prompt engineering that reduces token count as "throughput improvement"
- "This one weird sampler" — MinP, DRY, XTC etc. as *perf* claims (they're quality)
