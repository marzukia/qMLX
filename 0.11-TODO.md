## Phase 1 — SpinQuant rotation preprocessing (P0, HIGH EV / LOW risk)

Reference: arxiv 2405.16406 (Meta 2024) + multiple mature open-source implementations.

| Metric | Target |
|---|---|
| PPL gain | +1-3% on every 4bit hero alias |
| Decode tok/s cost | **0%** — reuses existing MLX int4 matmul kernel |
| Effort | 1-2 weeks (rotation matrix precompute + per-alias re-quant pass) |
| Risk | LOW |

### What it is
Learned rotation applied to weights BEFORE the existing MLX int4 quant. No new runtime kernel. The rotation redistributes activation magnitudes so more of the dynamic range fits under int4 quant with less loss.

### Deliverables
1. Rotation matrix precompute pipeline (script under `scripts/spinquant/`)
2. Per-alias re-quant pass — apply rotation, re-quantize, upload to mlx-community as `<alias>-SpinQuant-4bit`
3. New alias suffix `spinquant` in `aliases.json`; profile flag `spinquant_verified: true` on re-quantized variants
4. Bench delta: PPL on WikiText / HumanEval on all Phase-2 hero aliases (Qwen 3.6-27B, Gemma 4-12B, gpt-oss-20b)
5. If verified, promote SpinQuant-4bit to default under short alias (e.g. `qwen3.6-27b` → SpinQuant-4bit instead of vanilla-4bit)

### Blocking signal
Bench delta < +0.5% PPL on ≥2 of 3 hero aliases → abandon; the win doesn't justify user-facing new alias surface.

### One surprise signal to feed in
EXL3 4.15 bpw PPL beat MLX 4bit by 5% at 8B (from the 0.9 Stage B memo). Trellis-coded quant proved the quality direction; SpinQuant should reach the same target via rotation + MLX int4 kernel we already have.

## Phase 2 — W8A8 INT8 with fused Metal kernel (P1, MED risk)

Reference: standard int8 matmul path via Apple Silicon AMX on M3+ or NAX on M5. Currently we ship W4A16/bf16 — activations are bf16, not int8.

| Metric | Target |
|---|---|
| Decode | 1.3-1.5× on small models (E2B, gpt-oss-20b) |
| Quality | ~lossless (int8 quality bar historically higher than int4) |
| Effort | 2-3 weeks Metal kernel + NAX audit |
| Risk | MED |

### Blockers
1. **M5 NAX dtype-fallback audit** (Phase 4 below) — must confirm NAX exposes int8 dot product BEFORE investing kernel weeks.
2. Reasoning-model quality — W8A8 has historically been rough on long-form CoT emissions. Gate on gpt-oss-20b HumanEval + AIME25 numbers.

### Deliverables
1. Fused int8 matmul Metal kernel (or upstream via mlx-lm PR if the maintainers accept)
2. Per-alias re-quant pipeline (similar shape to Phase 1 SpinQuant)
3. Aliases: `<hero>-W8A8` variants for gpt-oss-20b + Gemma 4 E2B/E4B (small models where 1.3-1.5× is user-noticeable)
4. Reasoning quality guard — AIME25 + HumanEval must not regress > 2 points vs W4A16 baseline; else keep W8A8 variants advisory only

## Phase 3 — QuIP# / VPTQ 2-bit lattice-coded quant (P2, HIGH risk)

Reference: arxiv 2402.04396 (QuIP#) + arxiv 2404.14047 (VPTQ). **Lattice-coded** 2-bit — NOT trellis like EXL3 — designed for matmul-friendly codebook where dequant doesn't have to precede matmul.

| Metric | Target |
|---|---|
| Decode | 1.5-2× if memory-bandwidth bound |
| Fit | Unlocks 600B+ MoE (DeepSeek R2, Qwen 3.7-A-huge) on 96GB Mac |
| Effort | 3-4 weeks |
| Risk | **HIGH** — lattice-code throughput on M-series Metal untested; could hit the same cliff EXL3 did (1.77× decode regression) |

### Kill switch
If prototype Metal matmul kernel benches at <1.0× vs W4A16 baseline at 2-bit precision → this is EXL3 replay. Kill.

### Same use case EXL3 was meant for, done right
600B DeepSeek R2 / Qwen 3.7-A-huge on 96GB Mac Studio. Also GLM 5.2 REAP + 2-bit stack could put GLM 5.2 in a 32GB budget.

## Phase 4 — M5 NAX dtype-fallback audit (P1, blocking Phase 2)

### Deliverables
1. Telemetry counter — matmul shapes that miss NAX (fallback to CPU) surfaced in `/metrics` as `rapid_mlx_nax_matmul_fallback_total{shape}` label set
2. Audit report — inventory of every matmul shape our top 3 families hit; verify which land on NAX vs fallback path
3. If NAX exposes int8 dot product → Phase 2 unblocked
4. If NAX only exposes bf16/int4 → Phase 2 W8A8 downgraded to CPU AMX path (probably not worth the effort; may kill Phase 2)

### Deliverables independent of Phase 2
Even without Phase 2, this audit gives us a per-alias "NAX-effective yes/no" flag we can surface in `rapid-mlx info` — users know when they're getting the full M5 speed vs falling back.

## Don't do list (0.11-specific extension of 0.9's list)

Everything on 0.9-TODO's Don't-do stays valid. Additions specific to 0.11:

- **Don't ship SpinQuant + W8A8 simultaneously without a gate.** They're orthogonal but compound quality risk. Bench SpinQuant standalone first (Phase 1), then layer W8A8 on top.
- **Don't ship 2-bit lattice quant without a per-alias PPL gate.** EXL3's 2.5 bpw PPL collapsed at 1B (+56%). Small models need a floor test — if <1.5B model's PPL regresses > 20% at 2-bit, don't ship the alias for that size.
- **Don't repackage EXL3 findings as "new" work.** They're documented in the 0.9-TODO Stage A/B memos + `/tmp/exl3-stage-{a,b}.md`. Cite those.
